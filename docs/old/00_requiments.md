# **AI ローカルラジオアプリ (仮称) 要件定義書**

## **1\. 概要**

本アプリケーションは、ユーザーのローカル環境で動作する「生成AI駆動型ラジオプレーヤー」である。  
予め用意されたBGM楽曲と、リアルタイムに生成されるAIによるニューストークを交互に再生し、自分だけのラジオ体験を提供する。

### **コンセプト**

* **Mood & Flow**: ユーザーの気分に合わせた選曲とトーク。曲間にはラジオらしい自然な「間」を持たせる。  
* **Freshness**: 最新のニュース（RSS）をネタにした、その時だけのトーク。  
* **Low Latency**: BGM再生中に次を生成し、スムーズな進行を実現する。

## **2\. システムアーキテクチャ**

### **技術スタック**

* **Framework**: Wails (Go \+ Webview)  
* **Frontend**: React (UI, Audio Player)  
* **Backend (Go)**:  
  * ファイルシステム操作 (BGMスキャン, 設定保存)  
  * 外部API通信 (RSS取得, LLM推論, TTS生成)  
  * アプリケーション状態管理  
* **AI / API**:  
  * LLM: OpenAI API 互換 (openrouter/ollamaを想定) (トーク内容生成)  
  * TTS: Google Gemini API (音声生成)

### **データフロー概要**

1. **Frontend**: ユーザー操作（再生/停止/ジャンル変更）を受け付け、Backendへリクエスト。  
2. **Backend**:  
   * 選択されたジャンルフォルダから楽曲をランダム選出。  
   * BGM再生中に、登録された複数のRSSからランダムに記事を取得。  
   * LLMで台本作成、TTSで音声化。  
   * 生成された音声データへのパス、またはデータURIをFrontendへ返す。  
3. **Frontend**: 受け取ったオーディオデータ（BGMまたはトーク）をHTML5 Audio等で再生する。

## **3\. 機能要件**

### **3.1 再生機能 (Player Logic)**

* **サイクル再生**:  
  * 基本サイクル: \[BGM ×N\] \-\> \[AI Talk (約1分)\] \-\> (繰り返し)  
    * N はユーザーが設定可能（デフォルト: 3）  
  * 曲とトークの間、曲と曲の間には、ラジオ放送のような自然な「間（1〜3秒程度の無音）」を許容する（厳密なギャップレス/クロスフェードは必須としない）。  
* **ジャンル/ムード選択**:  
  * BGMルートディレクトリ直下のサブディレクトリを「ジャンル（またはムード）」として認識する。  
    * 例: BGM/Jazz/, BGM/Lo-Fi/, BGM/Morning/  
  * ユーザーは再生するジャンルを選択可能。選択されたサブフォルダ内の楽曲のみをシャッフル再生対象とする。

### **3.2 トーク生成機能 (AI Generator)**

* **ネタ元 (Source)**:  
  * **複数RSS対応**: ユーザーが登録した複数のRSS URLリストを保持する。  
  * **ランダムピックアップ**: トーク生成のタイミングで、登録リストの中からランダムに1つのRSSフィードを選び、さらにその中から最新記事をピックアップする。  
* **重複排除 (History)**:  
  * 過去にトークで使用した記事のURL（またはID）をローカルDB/JSONに保存。  
  * 記事選出時にヒストリーと照合し、既出の話題はスキップする。  
* **台本生成 (LLM)**:  
  * システムプロンプトにより「ラジオトーク風」の単一ペルソナを設定（当面は固定ペルソナ）。  
  * 入力: ニュース記事のテキスト。  
  * 出力: 約1分（約200〜300文字程度）の読み上げ用テキスト。  
* **音声合成 (TTS)**:  
  * LLMで生成されたテキストを Gemini TTS API に送信し、音声データ(MP3/WAV)を取得。

### **3.3 設定機能 (Configuration)**

* **API設定**:  
  * Gemini API Key の入力フォーム。  
  * 保存先: localStorage または Wails経由でローカルの設定ファイル（config.json等）。  
* **パス設定**:  
  * **BGMルートフォルダ**のパス指定。  
  * アプリはこのフォルダ内のサブフォルダをスキャンし、ジャンルリストを構築する。  
* **RSS設定**:  
  * 購読するRSS URLのリスト編集（追加・削除）。  
  * 初期設定としていくつかの主要ニュースサイトのRSSプリセットを用意しておくと親切。

### **3.4 状態保存**

* アプリ終了時に以下を保存し、次回起動時に復元する。  
  * APIキー  
  * 選択されたBGMルートパス  
  * **前回選択していたジャンル**  
  * 再生済み記事履歴 (History)

## **4\. UI/UX デザイン要件**

### **メイン画面 (Player View)**

* **現在再生中の情報**:  
  * ジャケット画像（あれば）または、ラジオ局ロゴ。  
  * 曲名 / アーティスト名。  
  * Talk中: トピックタイトル。  
* **操作系**:  
  * **ジャンル選択ドロップダウン**: 「Jazz」「Pop」「Work」などを切り替え。  
  * 再生/一時停止ボタン。  
  * スキップボタン。  
  * 音量調整（BGM音量 / Talk音量 を個別に調整可能）。

### **設定画面 (Settings Modal)**

* APIキー設定（Gemini）。  
* BGMルートフォルダ選択。  
* RSS URLリスト管理（追加/削除ボタン付きリスト）。
* トークまでのBGM曲数（N）の指定。  
* TTS（Gemini）設定（ボイスモデル/ボイス名の指定）。
* 音量（BGM / Talk）の指定。

## **5\. 技術的詳細・検討事項 (Deep Dive)**

### **5.1 ディレクトリ構成案**

UserAppData/  
  ├── config.json       (アプリ設定: BGMパス, RSSリスト, 前回ジャンル)  
  ├── history.json      (トーク済みURL履歴)  
  └── temp\_audio/       (生成されたTTSの一時保存場所。終了時削除)

UserBGMFolder/ (ユーザーが任意に指定)  
  ├── Jazz/  
  │   ├── song1.mp3  
  │   └── ...  
  ├── WorkBGM/  
  │   ├── songA.mp3  
  │   └── ...  
  └── ...

### **5.2 音声再生のハンドリング (Frontend Driven)**

* **Frontend (React)**:  
  * 全て Audio オブジェクトまたは \<audio\> タグで制御。  
  * onEnded イベントをフックして、「次の曲」または「トーク」をバックエンドに要求するトリガーとする。  
* **Backend (Go)**:  
  * Wailsの AssetServer 機能でローカルファイルへのアクセスを提供。  
  * フロントエンドからの「次をくれ」リクエストに対し、次の再生用URL（http://localhost:xxxx/assets/Jazz/song1.mp3 や生成されたTTSファイルのパス）を返す。

### **5.3 先読みと生成フロー制御 (Queue System)**

* 基本的に1つ先まで確保しておけば十分（厳密なギャップレス不要のため）。  
* 現在再生中の曲が終盤に差し掛かったら、次がトークの順番かチェックし、トーク生成を開始する程度の緩やかな先読みで実装する。

### **5.4 エラーハンドリング**

* **ネットワークエラー**: RSS取得やAPI接続失敗時。  
  * 失敗時はトークをスキップして次のBGMへ進む（サイレントフェイル）。  
* **BGM不足**: 指定ジャンルフォルダに曲がない場合。  
  * UI上で警告を出し、再生不可とする。

## **6\. 今後の拡張性 (Future Works)**

* 天気予報 API との連携。  
* 複数のトークペルソナ切り替え（元気なDJ、深夜便DJなど）。  
* BGMの自動解析によるムード分類（現在はフォルダ分け手動運用）。
